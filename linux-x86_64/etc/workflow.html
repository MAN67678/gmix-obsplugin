GMix — how it works

temporal frame-blending for high-fps games · Linux / Wayland / Vulkan
A game renders at hundreds of fps, but a stream is only 60fps — so most frames get thrown away and fast motion looks choppy. GMix captures every frame and blends all of them within each 1/60s into one, giving a clean 60fps video with real motion blur built from actual rendered frames (not a fake shader).
1

Game rendersosu!

osu! draws frames as fast as the GPU allows — often 300–1000 fps. Each frame holds real positional data about the cursor and notes between the 60fps "ticks" a normal recording would keep.

↓every presented frame
2

Capture layerVkLayer_GMIX (inside osu!)

A Vulkan layer hooks the game's frame-present. On each frame it copies the finished image and exports it as a shared GPU buffer — no CPU copy, no encode. Frames never leave the GPU.

zero-copy: shared memory + sync via fd over a socket
injected automatically — game is unmodified
↓shared GPU frame + timestamp
3

Ring buffergmix (separate process)

GMix imports the shared frames into a rolling buffer of the most recent ones, always keeping the freshest (old backlog is dropped so latency stays low and constant).

↓frames from the last 1/60s
4

Shutter blendcompute shader, 60 Hz

60 times a second, a GPU compute shader takes every frame captured within the last 1/60s and blends them per-pixel — exactly like a real camera shutter. Fast motion smears into a smooth trail; anything still stays sharp.

↓one clean blended frame
5

Output windowWayland, 60 fps

The blended result is shown in a normal 60fps window. OBS captures it like any other source — what goes to the stream is a mathematically real average of all the motion that actually happened.
